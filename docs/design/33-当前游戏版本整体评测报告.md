# 33-当前游戏版本整体评测报告

> 评测依据：`30-整体评测方案.md`  
> 评测对象：当前 `prototype/hifi` 高保真游戏版本  
> 评测日期：2026-06-25

## 1. 总结

| 结论 | 判断 |
|---|---|
| 当前版本可以继续开发 | L1 全量测试通过，核心引擎没有崩溃红线 |
| 当前版本还不适合进入大规模数值调平 | L2 统计体系尚未工程化，且资源、合法性、机制活性分布已经暴露明显问题 |
| 游戏目标感比之前更清楚 | 御前会议、局势入口、资源账本、外交分组让玩家更容易知道“现在要处理什么” |
| 最大问题不是功能少，而是系统压力不够稳定 | 裁断长期为 0、从属关系 0 次触发、建设与科技在 40 季样本中几乎不活跃 |

一句话判断：**当前版本从“能跑的原型”进化到了“可评测的策略框架”，但还不是“稳定好玩的策略游戏”。**

| 项目 | 结果 |
|---|---|

L1 结论：**当前版本没有阻断开发的正确性问题。**

## 3. L2 批量模拟评测

说明：30 号方案里的 `tools/sim` 还没有落地，所以本次使用临时 Node 统计脚本跑当前 hifi 引擎，不写入项目。样本为：

### 3.2 资源与国家强度

## 5. 分系统评估

| 系统 | 当前状态 | 主要问题 | 建议 |
|---|---|---|---|
| 地图 | 已是主界面核心，地块、模式、详情基本可用 | 地形横幅 404 | 先修路径 |
| 国家 | 国家、领导人、政体、阶层都已接入 | 弱国早期合法性与资源坠落 | 给小国保底操作空间，不是直接送钱 |
| 外交 | 条约活跃，列表可读性改善 | 从属关系 0 次触发 | 做“可从属目标”和“宗主收益”引导 |
| 战争 | 状态机稳定，测试覆盖较好 | 40 季几乎固定 1 场战争，局势长期 open_war | 让战争目标、停战、阶段评估更明显 |
| 经济 | 资源账本提高可解释性 | 金钱跨度极大，建设不活跃 | 先补建设触发，再调资源曲线 |
| 发展 | 科技框架存在 | 40 季无采纳 | 用更长样本评估，不急着调 |
| 御前会议 | 有待办和直达 | 阻塞裁断 0 | 增加真正需要玩家二选一的压力点 |
| UI | HUD、抽屉、弹窗结构清楚 | 抽屉与局势弹窗会同时占屏 | 打开局势/国家弹窗时关闭系统抽屉 |

## 6. 优先修复清单

| 顺序 | 问题 | 为什么先做 | 修复方向 |
|---:|---|---|---|
| 1 | 地形横幅 404 | 明确错误，成本低，影响观感 | 修正 `map.js` 和 CSS 中 `../../assets/...` 的相对路径 |
| 2 | 局势弹窗打开后系统抽屉仍占屏 | 影响核心操作体验 | 打开 modal 类主界面时统一关闭系统抽屉和 active 按钮 |
| 3 | 从属机制 0 次触发 | 已设计但没形成玩法 | 增加附庸/朝贡可行性提示、AI 尝试、玩家收益预览 |
| 4 | 阻塞裁断 0 | 御前会议缺少真正决策压力 | 把局势、财政、阶层冲突转为少量必须处理的裁断 |
| 5 | 小国资源和合法性坠落 | 影响可玩国家范围 | 给小国可主动选择的恢复手段，而不是静态补贴 |
| 6 | 正式落地 `tools/sim` | 没有它就无法稳定做数值回归 | 按 30 号方案补 `run/metrics/report/baselines` |

## 7. 不建议现在做的事

| 不建议 | 原因 |
|---|---|
| 直接大幅调整所有资源产出 | 当前还没有正式 L2 基线，容易越调越乱 |
| 继续堆新系统 | 现有从属、建设、裁断还没活起来 |
| 把弱国直接补成强国 | 会破坏历史差异；应该补“可操作选择”，不是补结果 |
| 把所有待办都变成阻塞 | 会造成频繁打断；阻塞只留给真正需要玩家裁决的事项 |

## 8. 最终判断

当前版本已经达到：

| 阶段 | 是否达到 |
|---|---|
| 可运行 | 是 |
| 可测试 | 是 |
| 可评测 | 初步达到 |
| 可长线游玩 | 未达到 |
| 可数值调平 | 暂未达到 |

下一步最稳的路线是：**先修两个 UI 明确问题，再补从属/裁断/建设活性，最后落地 `tools/sim` 做正式数值回归。**
