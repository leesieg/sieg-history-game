# 28 - 局势系统优化方案（基于 27 号试玩报告）

## 目标

把局势系统从「阶段在转、信息很多、但战争是假进程、行动会报错、目标不推进」修成**一局能玩、能赢能输、有取舍、有节奏的法兰西战役**。本方案严格对齐 27 号报告的五项优先级与七条验收标准，每条改动都落到 `prototype/hifi/scripts/` 的具体函数。

设计原则不变（CLAUDE.md 核心循环）：局势是挂在「战况流」上的转折层阀门——**它读真实战争产出阶段，阶段再回灌资源/疲惫/合法性**。当前问题的总根源是：阶段被一个**与真实战争无关的时钟**驱动，于是转折层空转、回灌也跟着空转。

## 二、优化设计（按报告优先级）

### P1 — 修复局势决议按钮可用态（规则可信度，先修）

**单一真相源**：把「真实前置」从 `executeStruggleAction` 抽进 `PHASE_ACTIONS`，让 gate 和执行器共用同一套前置。

```js
// struggle.js — 每个操作声明一个 precheck，返回 {ok, reason}
const PHASE_ACTIONS = {
  press_claim:   { label:"提王位主张", phase:"standoff", involvement:"principal",
                   precheck: (w,p,s) => atWarWithOpponent(w,p,s)
                     ? { ok:false, reason:`与${opponentOf(s,p)}已交战，无需再提主张` } : { ok:true } },
  muster_battle: { label:"决战集结", phase:"open_war", involvement:"principal",
                   precheck: (w,p) => hasCity(w,p) ? {ok:true} : {ok:false, reason:"没有可集结的城市"} },
  favorable_truce:{label:"有利停战谈判", phase:"truce", involvement:"principal",
                   precheck: (w,p,s) => principalWar(w,s) ? {ok:true} : {ok:false, reason:"目前没有与对手的战争可议和"} },
  pick_side:     { label:"选边支持", phase:null, involvement:"interloper", precheck: ()=>({ok:true}) },
};
```

- `phaseActionGate` 增加第三步：通过 role + phase 后调 `action.precheck`，失败抛同一条中文。
- `executeStruggleAction`（main.js）删除内联的 areAtWar / war 存在判断，改为「先 `phaseActionGate`（已含 precheck）→ 直接委托引擎」。
- `decisionsFor` 不变（已经 `try{phaseActionGate}`），自动获得 precheck 结果。

**结果**：面板里亮着的决议 == 点了一定能执行；置灰的附上同一条原因。**验收①达成**。

### P2 — 局势阶段与真实战争同步（系统是否成立的核心）

**核心改动：废掉盲目时钟，让阶段由真实战争状态驱动。** 改写 `processStruggles` 的默认诱因：

```js
function processStruggles(world) {
  for (const s of activeStruggles(world)) {
    if (s.phase === "resolution") continue;
    applyCatalysts(world, s);                 // 战况诱因（保留）
    const war = principalWar(world, s);
    if (war) {
      addCatalyst(s, "open_war", 1);          // 有真实战争 → 维持/推向鏖战
    } else if (s.phase === "open_war") {
      addCatalyst(s, "truce", 2);             // 在鏖战但战争已结束 → 推向议和（每季持续推，不只转换当季）
    } else if (s.phase === "truce") {
      addCatalyst(s, "standoff", 1);          // 议和恢复期 → 回到冷对峙
    }
    // standoff 无战争时：不再自动推向 open_war。改为累计再开战压力，先预警再开战
    updateWarPressure(world, s);
    flipIfReady(world, s);
  }
}
```

**再开战压力（替代粗暴自动逼战）**：standoff 持续期间不直接宣战，而是累计 `warPressure`。压力来自边境冲突、停战到期、英格兰战略利益、法兰西未完成“收复争议领地”等因素。压力达到阈值后先进入“战云密布”预警，再给玩家 1–2 季准备窗口；准备窗口结束后，如果法英仍未达成外交缓和，才由非玩家当事国发起再开战。

```js
function updateWarPressure(world, s) {
  if (s.phase !== "standoff") return;
  if (principalWar(world, s)) return;

  s.warPressure ||= 0;
  const principals = principalsOf(s);
  const [france, england] = ["法兰西王国", "英格兰王国"];
  const truceActive = window.HIFI_WARFARE_ENGINE.underTruce(world, france, england);

  if (!truceActive) s.warPressure += 1;                 // 停战保护结束，压力自然回升
  if (!missionDone(world, france, "reclaim-disputed")) s.warPressure += 1;
  if (borderTension(world, s) > 0) s.warPressure += 1;

  if (s.warPressure >= 6 && !s.warningTurn) {
    s.warningTurn = world.turn;
    logStruggleEvent(world, `${s.label}战云密布，双方开始重新集结`);
    world.pendingStruggleWarning = { key: s.key, label: s.label, kind: "war_pressure" };
  }

  const preparationDone = s.warningTurn && world.turn - s.warningTurn >= 2;
  if (s.warPressure >= 8 && preparationDone && !truceActive) {
    const ai = principals.find(name => name !== world.playerPolity) || england;
    window.HIFI_WARFARE_ENGINE.declareWarOn(world, ai, world.playerPolity, `${s.label}·重燃`);
    s.warPressure = 0;
    s.warningTurn = null;
    addCatalyst(s, "open_war", definition(s.key).flipThreshold);
    logStruggleEvent(world, `${s.label}重燃战火，${ai}再度兴兵`);
  }
}
```

这让百年战争呈现历史上的「打—停—再打」节奏，同时保留玩家准备窗口。玩家被动不处理，战争会回来；玩家主动处理，可以通过外交、备战、议和争取更好的再开战条件。

**阶段命名诚实化（无战争不显示「鏖战」）**：`struggleSummary` 增加 `warActive`，UI 据此改写显示名：

| 内部 phase | 有真实战争 | 无真实战争 |
|---|---|---|
| standoff | 对峙 | 对峙 |
| open_war | **鏖战** | **备战**（mustering，不显示鏖战） |
| truce | 疲惫议和 | 疲惫议和 |

在「备战」显示态下，置灰需要真实战争的决议，推荐改成「集结备战 / 拉拢盟友 / 降低再开战压力」。**验收②达成（无真实战争不显示鏖战、不放可执行的战争型决议）**。

**使命推进补链**：仅靠再开战还不够。局势面板必须把“收复争议领地”拆成可操作链：

| 阶段 | 推荐动作 | 目标 |
|---|---|---|
| 对峙 / 备战 | 集结鲁昂主力、补员、拉拢苏格兰 | 准备下一轮战争 |
| 鏖战 | 标记目标地、规划军团路线、围攻目标 | 推进战争分数和占领 |
| 疲惫议和 | 预览和平条款、判断是否停战 | 完成或延后使命 |

该链路要进入 `recommendations` 和局势面板，不能只停留在阶段文字。

### P3 — 40 回合阶段性结算（十年战局评估）

新增**非终结**的十年评估，复用季度总结弹窗，不 resolve 局势：

```js
// struggle.js
function reviewStruggles(world) {
  const cal = window.HIFI_WORLD_ENGINE?.calendarForTurn;
  for (const s of activeStruggles(world)) {
    const year = cal ? cal(world.turn).year : null;
    const elapsed = world.turn - s.startedTurn;
    if (elapsed < 40 || elapsed % 40 !== 0) continue;       // 每 40 季评估一次，首次约 1347 年
    if (s.lastReviewTurn === world.turn) continue;
    s.lastReviewTurn = world.turn;
    const stages = missionStages(world, "法兰西王国");
    const done = stages.filter(x => x.status === "已完成").length;
    const war = principalWar(world, s);
    const exhaustion = world.countries["法兰西王国"]?.warfare?.warExhaustion || 0;
    const advantage = done*2 + (war ? Math.sign(war.score) : 0) - Math.floor(exhaustion/20);
    const verdict = advantage >= 3 ? "优势" : advantage <= -1 ? "劣势" : "胶着";
    // 轻量回灌（非终局）：优势 +合法性 / 劣势 -合法性并加压
    const fr = world.countries["法兰西王国"];
    if (fr) fr.legitimacy = clamp(fr.legitimacy + (verdict==="优势"?4:verdict==="劣势"?-4:0));
    world.pendingStruggleReview = { label:s.label, year, turn:world.turn, verdict, done, total:stages.length, exhaustion };
  }
}
```

- 在 `processHistory` 的 `settleStruggles` 之前调用 `reviewStruggles`。
- dialogs 增 `renderStruggleReview`，沿用 `seasonSummaryModal`：标题「{1347}年 · 十年战局评估」，正文列：使命进度 done/total、当前战争分数、战争疲惫、判定（优势/劣势/胶着）、本次合法性奖惩。
- 第 40 季左右必触发一次；后续每 40 季触发一次。不要用 `year % 10`，否则 1347 年不会触发。**验收⑦达成**。

### P4 — 强化战争疲惫与资源消耗（让长期战争有取舍）

**疲惫接入鏖战阶段**（不再只靠占领）。在 `applyStruggleWarPressure`（history.js）里，对 `open_war` 且有真实战争的当事国每季累加疲惫：

```js
if (struggle.phase === "open_war" && principalWar(world, struggle)) {
  country.warfare.warExhaustion = Math.min(100, (country.warfare.warExhaustion||0)
    + 1 + Math.floor((world.turn - struggle.phaseSinceTurn)/4));   // 鏖战越久涨越快
}
```

**资源消耗改为按军队规模 + 战争强度 + 补给状态计算**。不能按库存百分比扣，因为那会惩罚“攒资源”的玩家，也不符合战争成本来源。消耗应来自军团、人力、补给线、战争阶段。

| 阶段 | 每季消耗 |
|---|---|
| open_war（有战） | 按参战军团兵力扣粮食；按职业/常备/佣兵权重扣军需；补给不足追加损耗 |
| truce | 按战争疲惫和伤兵规模扣少量金钱，表示抚恤与重整 |
| standoff（备战） | 若 `warPressure >= 6`，按已动员军团规模扣少量军需 |

建议初始公式：

```js
const engagedSoldiers = principalArmies(world, polity).reduce((sum, army) => sum + armySoldiers(army), 0);
const intensity = struggle.phase === "open_war" && principalWar(world, struggle) ? 1 : 0;
const foodCost = Math.ceil(engagedSoldiers / 1000 * 1.5 * intensity);
const militaryCost = Math.ceil(engagedSoldiers / 1000 * serviceWeight(army) * intensity);
const moneyCost = struggle.phase === "truce"
  ? Math.ceil((country.warfare?.warExhaustion || 0) / 5)
  : 0;
```

所有消耗写进 `lastReport.war`，经 `quarterLedger`（已支持 war 段）展示来源，可解释、不与黑死病/维护叠算。

**疲惫回灌扩面**（warfare.js `processWarExhaustion`，当前只扣合法性 L561-563）：疲惫 >20 时额外按比例压低本季产出（接 economy 的 lastReport 或在 applyStruggleWarPressure 内追加一笔产出折损），并降阶层满意。**验收④⑤达成（资源不无脑膨胀、鏖战期疲惫必涨且反作用国家状态）**。

### P5 — 右侧待办分组（主线 / 阻塞裁断 / 普通机会）

`issues()` 已有 `blocking`/`kind`，补一个 `tier` 标签并在渲染端分组：

```js
// kind→tier 映射
// election/event(裁断)/situation爆发 → "blocking"
// struggle/war/mission                → "mainline"
// council预警/transition/建设机会        → "opportunity"
```

- 给局势相关待办加一条 `{tier:"mainline"}`：当前阶段名 + 一条保底推荐，点击直达局势面板。
- 黑死病满足“首次爆发 / 首都圈受影响 / 人口损失超过阈值”之一时升级为 `blocking`。
- 渲染端按 `blocking → mainline → opportunity` 三组带小标题输出，每组内排序。**验收（待办优先级清晰）**。

## 三、次要优化（报告其余项）

- **推荐必须不为空（报告建议 #5）**：`recommendations` 增加「每阶段保底表」，无论启发式是否触发都先给该阶段 2 条可执行/可跳转建议：
  - 对峙/备战 → 「集结备战」「拉拢苏格兰牵制英格兰」「降低再开战压力」
  - 鏖战 → 「集结主力决战」「增援边境争议地」「围攻目标地」
  - 疲惫议和 → 「发起有利停战谈判」「休整恢复疲惫」「准备下一轮对峙」
  每条带 `actionId` 或 `panel` 跳转，面板可点。**验收①的「推荐稳定非空」**。
- **黑死病反馈强化（报告建议 #8，验收⑥）**：爆发期事件文案显式给出**人口损失数 / 救济花费 / 产出下降 / 预计恢复季数**（数据 `processSituations` 已算，只是没展示）；只在首次爆发、首都圈受影响、或人口损失超过阈值时进入 `blocking` 队列，避免每次脉冲都打断长线推进。
- **UI 行动少于信息（报告 #10）**：`renderStrugglePanel` 把「推荐下一步（≥2 条可点）」提到决议区上方，信息段（结局预览）折叠次要，让面板像操作台不像展示板。

## 四、验收对照

| 报告验收项 | 本方案达成途径 |
