# 30-整体评测方案

> 设计依据：业界策略游戏测试调研（四层分级法）+ 本项目现状（`tests/*.test.cjs` 确定性测试 + 27/29 号一次性试玩报告）。
> 目标：把现在零散的"写个长跑脚本跑一次、手写一份报告"固化成**可复现、有指标、能回归**的评测体系，让"局势系统优化方案（28 号）"这类改动可以用数据验证前后差异，而不是靠人眼观察。

## 1. 为什么需要这份方案

当前测试落在两端，中间是空的：

| 已有 | 缺口 |
|---|---|
| 第一层：`hifi-*` 确定性测试（规则函数、结构、UI 接线），抓"引擎报不报错" | **第二层：模拟驱动的平衡评测**——没有固定指标、没有可复现的批量自对弈产物 |
| 第三层手工版：27/29 号试玩报告（人/LLM 跑一遍写结论） | 报告是一次性的，换个版本要重新手写，无法 diff、无法回归 |

策略游戏的"好不好"是**涌现属性**：单元测试只能断言守恒律，断不了"军事路线是否过强""某机制是否空转"。这些只能从**大量自对弈的统计分布**里读。本项目的核心循环（`advanceQuarter` 8 步全自动、无玩家也能自转）天然适合无人自对弈——这是第二层评测的地基，必须用起来。

## 2. 四层评测框架（业界分级 → 本项目落点）

业界按"确定性"从硬到软分四层，每层产出物不同。本项目对号入座如下：

| 层 | 业界做法 | 产出形态 | 本项目落点 | 现状 |
|---|---|---|---|---|
| L1 正确性 | 单元/集成/不变量/属性测试 | pass / fail | `tests/hifi-*.test.cjs` + 新增不变量长跑 | 大体有，缺不变量长跑 |
| L2 平衡评测 | 蒙特卡洛自对弈、自动平衡、受限对弈 | **分布与曲线**（非 pass/fail） | 新增 `tools/sim/` 批量自对弈 + 指标导出 | **空白，本方案重点** |
| L3 体验评估 | 结构化 playtest、启发式审查、LLM 试玩 | 定性报告 + 卡点清单 | 27/29 号报告，流程化为 `tools/playtest/` | 一次性，待固化 |
| L4 运营遥测 | 留存/漏斗/线上平衡看板 | 实时指标 | 原型阶段不做，仅预留埋点约定 | 暂缓 |

> [!note] 关键认知
> L1 产出 **pass/fail**，L2 产出 **分布**。不要把平衡性写成 `assert`——"军事路线胜率 = 40%" 不该红/绿，它该是一条随版本变化的曲线，配一个**健康区间**（如 30%–55%），越界才告警。把平衡硬断言会导致测试天天因为正常的数值波动而红。

---

## 3. L1：确定性正确性（守住底线）

### 3.1 现有保留
`hifi-world/turn/politics/economy/diplomacy/warfare/history/...` 全部保留，继续作为"引擎不报错、结构合法"的红线。

### 3.2 新增：不变量长跑测试 `tests/hifi-invariants.test.cjs`
固定种子，让世界空跑 200+ 季，每季断言**守恒律**而非具体数值。这是性价比最高的补强，能直接抓住 29 号暴露的"自动宣战崩溃""资源膨胀"类系统性问题。

不变量清单（断言"永远成立"，越界即 fail）：

| 不变量 | 断言 |
|---|---|
| 资源非负 | 所有国家 `food/money/military` 为有限数，不出现 `NaN/Infinity`；库存下限符合规则 |
| 回合单调 | `world.turn` 每季严格 +1，年份/季节字段同步 |
| 战争状态机自洽 | 不对已交战双方再次宣战；不在停战协定期内宣战（**29 号崩溃根因**） |
| power 守恒 | 各阵营 power 之和在合理区间，单国不溢出 |
| 阶段合法 | 局势 phase 只在定义的状态集内迁移，不越界 |
| 不抛错 | 200 季全程 `advanceQuarter` 不 throw |

> [!tip]
> 不变量测试和 28 号"P1 决议单一真相源""P4 比例消耗"是互补的：28 号修因，本测试守果。修完 28 号后，这个长跑应能从"中途崩溃"变成"跑满 200 季绿灯"。

### 3.3 可选：属性测试
用随机生成的合法操作序列轰炸引擎，只校验 3.2 的不变量。能发现人想不到的边界组合。原型阶段优先级低于 L2，列为储备。

## 4. L2：模拟驱动的平衡评测（本方案核心）

### 4.1 产物与目录
新增 `tools/sim/`（不进 `tests/`，因为它产出数据不是断言）：

```
tools/sim/
  run.cjs          # 批量自对弈：复用 hifi 引擎，N 局 × M 季，导出原始数据
  metrics.cjs      # 从原始数据计算指标分布
  report.cjs       # 生成 markdown 报告（对齐 29 号表格格式）
  baselines/       # 各版本指标快照，用于 diff
```

复用 `hifi-longrun.test.cjs` 已验证的引擎加载方式（`vm.runInNewContext` 装载 `scripts/*.js`）。每局可指定**策略画像**（passive/balanced/military/economy/diplomacy，沿用 29 号分路线口径），模拟不同玩家倾向。

> [!warning] 时间预算
> 29 号踩过坑：浏览器跑 100 局超 5 分钟工具上限。L2 一律走**引擎层**（纯 Node，无 DOM），浏览器只做 L3 巡检。引擎层 100 局 × 40 季应在分钟级完成。

### 4.2 指标体系（沿用并固化 29 号口径）
对每局收集，跨局统计 min/25%/中位/75%/max/均值。分三类：

**A. 稳定性指标（越界 = 红线，回流 L1）**
- 完成率：跑满目标季数的局数占比（29 号基线 92/100，目标 → 100/100）
- 崩溃分类：按异常原因聚类（自动宣战、空指针……）
- 长样本完成率：100 季 / 百年跨度样本能否跑满

**B. 平衡指标（看分布是否在健康区间）**
- 分路线胜率/终局强度（military 不应碾压其余线）
- 资源分布断裂度：金钱 max/min 跨度（29 号 0→3096，过大）
- 合法性归零率（29 号 25%，惩罚过重）
- 战争疲惫形态：是否渐进累积，而非"0 或 100 断裂"
- 思想点离散度（29 号 92 局全 60，过于固定）

**C. 机制活性指标（抓空转机制，对齐 CLAUDE.md 总纲）**
- 每个机制阀门/决议被触发的频次：长期 0 次 = 疑似空转或假按钮
- "改它是否改变输出分布"：对关键参数做敏感性抽样，零敏感 = 空转
- 阻塞裁断数：长期 0 说明事件压力被自动消化、偏展示化（29 号现象）

### 4.3 健康区间（示例，随版本校准，存 `baselines/`）

| 指标 | 健康区间 | 越界含义 |
|---|---|---|
| 完成率 | = 100% | 有崩溃 → L1 红线 |
| 各路线相对终局强度 | 最强/最弱 ≤ 2× | 存在统治性路线 |
| 合法性归零率 | < 10% | 惩罚过重 |
| 战争疲惫"0 或 100"占比 | < 20% | 不是渐进压力 |
| 关键机制触发率 | > 0 | 空转 / 假按钮 |

> [!note]
> 区间数值是**起点**，不是真理。第一次跑出基线后人工评审，把"可接受的现状"定为区间，之后改动看是否偏离。这正是业界"受限对弈 + 自动平衡"的轻量替代：先量化、再调参、再回归。

### 4.4 与 28 号优化方案的接口
28 号的 5 项优化（P1 单一真相源 / P2 战争驱动阶段 / P3 十年评估 / P4 比例消耗 / P5 待办分层）改完后，**用同一套 `tools/sim` 前后各跑一次，diff 指标**：
- P4 比例消耗 → B 类"资源分布断裂度"应收窄
- P2 战争驱动阶段 → 阶段迁移不再被时钟机械翻转，C 类阶段活性应与真实战争事件相关
- 这样每项优化都有可量化的"改前/改后"证据，而非口头声明。

## 5. L3：体验评估（把 27/29 号流程化）

### 5.1 引擎层批量试玩报告
`tools/sim/report.cjs` 直接产出对齐 29 号格式的 markdown（数值表 + 分路线表 + 问题清单），存 `docs/design/` 续编号。**从手写变成一键生成**，换版本重跑即可 diff。

### 5.2 浏览器按钮巡检
保留 27/29 号的浏览器巡检：逐类点击按钮/抽屉/tab/地图工具，核对——
- 假按钮：条件不足是否禁用（29 号待办第 2 项）
- modal 是否互斥（29 号待办第 3 项）
- 待办是否直达对应系统（29 号待办第 4 项）

可半自动化为脚本断言"条件不足的入口必须 disabled"，纳入 `hifi-ui-smoke` 体系。

### 5.3 LLM-as-playtester
把"LLM 跑一遍写报告"固化为可重复 prompt + 输入（喂 `tools/sim` 的原始数据 + 截图），让 LLM 聚焦定性卡点与目标感（导师指引滞后、中期目标弱等 29 号定性问题），补模拟读不出的"好不好玩"。

## 6. L4：运营遥测（原型阶段仅预留）

不实装。仅约定：未来若上线，埋点应覆盖**动作漏斗 / 难度尖峰（失败率突增）/ 各路线留存**，复用 L2 已定义的指标口径，做到原型期与运营期指标同源。

## 7. 评测节奏（什么时候跑什么）

| 时机 | 跑哪层 | 红线 |
|---|---|---|
| 每次提交前 | L1 全量（`for f in tests/*.test.cjs; do node "$f"; done`） | 必须全绿 |
| 改动涉及引擎/数值/局势 | L1 + L2 批量自对弈，diff `baselines/` | 完成率 100%，平衡指标不显著恶化 |
| 阶段性里程碑（如 28 号落地完成） | L1 + L2 + L3 全套，产出续编号报告 | 健康区间达标，定性卡点收敛 |

| 任务 | 文件 | 内容 | 完成标准 |
|---|---|---|---|

| T2 自对弈引擎 | `tools/sim/run.cjs` | 复用 longrun 加载方式，N×M 局，按路线画像，导出原始数据 | 100 局 × 40 季分钟级完成、产出 JSON |
