# 欧陆风云5开发日志 1-7 总结与写作风格

> [!note] 范围说明
> 本文沉淀自 B 站文集《EU5开发日志》前 7 篇的标题、摘要、可访问备份片段和前次整理结果。B 站正文页曾触发验证码与频控，因此本文不把无法完整核验的正文细节写成逐字引用。

## 篇目总结

| 篇目          | 主题       | 核心内容                                                                          | 设计关键词         |
| ----------- | -------- | ----------------------------------------------------------------------------- | ------------- |
| #1 人口与动态世界  | POP 是底层  | 时间线横跨 1337-1837。世界不再只由抽象省份数值构成，而由 POP、劳动、身份和环境变化共同驱动。人口影响生产、税收、兵源、宗教、文化和社会结构。 | 动态世界、POP、因果反馈 |
| #2 政府、政治与阶层 | 国家不是一个按钮 | 政府、政治和阶层决定国家如何运行。玩家要平衡社会价值、议会、阶层诉求、合法性、改革和政治冲突。                               | 政府、阶层、合法性、改革  |
| #3 贸易与经济    | 经济是网络    | 每个地点生产原材料，建筑和生产链把资源转成财富。市场、贸易、供需、繁荣度互相影响，经济不是单纯收钱。                            | 生产、建筑、市场、贸易   |
| #4 探索与殖民    | 野心、风险、回报 | 15-19 世纪的远洋探索、殖民扩张和全球贸易网络。探索带来新资源、新市场和新战略空间，也带来成本、疾病、距离和治理压力。                 | 探索、殖民、全球贸易、风险 |
| #5 宗教与文化    | POP 的身份  | 人口不仅有数量，还有宗教、文化和语言。身份影响需求、归属、忠诚、融合、冲突和国家稳定。                                   | 宗教、文化、语言、认同   |
| #6 军事与战争    | 军队来自社会   | 战争从中世纪征召兵和骑士，逐步走向火药军队、职业军队和大规模征兵。军队来自真实人口，后勤、补给和战争消耗会反噬国家内部。                  | 征兵、后勤、人口、战争代价 |
| #7 外交与国际关系  | 战争之外的权力  | 外交决定战争前后的世界。联盟、敌对、帝国政治、谈判、战争目标和国际格局共同构成战略空间。神圣罗马帝国内部多国关系是复杂外交网络的典型场景。         | 外交、联盟、神罗、战争目标 |

## 共同设计逻辑

这 7 篇开发日志并不是孤立介绍功能，而是在搭建一套统一的大战略底层。

| 系统 | 它回答的问题 | 与其他系统的关系 |
|---|---|---|
| POP | 世界由谁组成？ | 连接经济、军事、宗教、文化、阶层 |
| 政府与阶层 | 国家如何统治？ | 连接改革、合法性、税收、社会冲突 |
| 经济与贸易 | 财富如何产生和流动？ | 连接地点、建筑、市场、POP、殖民 |
| 探索与殖民 | 国家如何走出本土？ | 连接海军、市场、疾病、海外治理 |
| 宗教与文化 | 人口为什么服从或反抗？ | 连接整合、叛乱、身份和合法性 |
| 军事与战争 | 国家如何使用暴力？ | 连接人口、后勤、财政、外交 |
| 外交 | 国家如何不用战争改变世界？ | 连接联盟、战争目标、和平和国际组织 |

核心思想可以概括为：

**EU5 想把国家、人口、经济、战争、外交都接到同一套动态历史模拟底层上。**

## 写作风格

| 写作习惯 | 具体表现 | 可迁移规则 |
|---|---|---|
| 先讲大主题 | 每篇开头先说本周聊什么，如人口、政府、经济、战争、外交 | 不要先列按钮和数值，先告诉读者这个系统为什么重要 |
| 历史背景先行 | 讲探索会先讲 15-19 世纪航海，讲战争会先讲军制演变 | 机制要从时代压力中自然长出来 |
| 强调动态世界 | 高频表达世界会变化、选择会产生后果、系统会互相影响 | 写日志时要写清楚“玩家做 A，会让 B 和 C 变化” |
| 第二人称代入 | 经常使用“你的国家”“你将面临”这类表达 | 用“你”制造代入，用“玩家”保持设计说明清楚 |
| 大系统拆小模块 | 一篇一个主题，再拆成生产、市场、建筑、阶层、后勤等子系统 | 每篇只讲一个核心系统，但要说明它如何连接其他系统 |
| 机制服务叙事 | 不是直接讲公式，而是讲为什么这个机制能表达历史 | 数值可以少，因果链必须清楚 |
| 宣传感但不空 | 会用宏大词汇，但最终落到 POP、市场、后勤、外交等具体机制 | 可以有气势，但每段都要有可落地的规则或玩家选择 |
| 承上启下 | 后续日志会回扣前面的 POP、经济、战争等基础 | 系列日志要不断串联前文，不要每篇从零开始 |

## 常用结构

| 段落 | 写法 |
|---|---|
| 开场 | “大家好，今天我们聊 X。” |
| 主题定义 | “X 不是装饰，而是国家运行的核心。” |
| 历史背景 | 用时代变化说明为什么需要这个系统 |
| 设计目标 | 用一句加粗句子概括系统原则 |
| 机制拆解 | 拆成 3-6 个玩家能理解的子模块 |
| 玩家取舍 | 明确收益、代价、风险和长期后果 |
| 系统联动 | 说明如何影响人口、经济、战争、外交、地图、阶层 |
| 当前状态 | 说明已实现、正在验证或后续计划 |
| 结尾 | 用一句设计原则收束，并预告下一篇 |

## 对《帝国的代价》的写作要求

后续本项目开发日志应遵守以下规则。

| 要求 | 说明 |
|---|---|
| 先讲时代和矛盾 | 先说明 1337 年、国家形成、黑死病、战争、贸易、阶层等背景 |
| 再讲设计目标 | 每篇必须有一句加粗设计原则 |
| 机制必须有代价 | 扩张、征兵、改革、建市场、殖民、外交都不能只有收益 |
| 必须写系统联动 | 地图、POP、经济、阶层、战争、外交至少联动两个 |
| 适合微信小游戏 | 规则可以中重，但表达要清楚，UI 和结算由数字载体承担 |
| 不写空泛宣传 | 不写“海量玩法”“极致策略”等没有机制支撑的话 |

## 一句话沉淀

**Paradox 风格开发日志的重点不是介绍功能，而是解释一个历史系统为什么存在、玩家如何使用它、以及它会让世界付出什么代价。**
